### 2. What Happens Inside the Computer? (Execution Flow & Processor Walkthrough)

When you run those commands, a lot of fascinating coordination happens between the software, the operating system (OS), and your Mac's processor (CPU). Here is the step-by-step breakdown:

#### Step 1: Compilation (`javac`)
* **What happens**: The processor executes the Java Compiler (`javac`). It reads your human-readable `.java` text files and translates them into `.class` files.
* **Under the hood**: The CPU parses the text, checks for syntax rules, and generates **Bytecode**—a special set of instructions that any Java Virtual Machine (JVM) can understand. It writes these bytes to the `out` folder on your hard drive.

#### Step 2: Starting the Server (`java ChatServer`)
* **What happens**: Your OS launches a new **JVM process**. The JVM acts like a simulated mini-computer running inside your Mac.
* **Loading**: The JVM's ClassLoader reads `ChatServer.class` from the hard drive into your computer's RAM.
* **Execution**: The processor locates the `public static void main` method. Because it's Java bytecode, the JVM uses a component called the "Just-In-Time (JIT) Compiler" to instantly translate the bytecode into native Apple Silicon/Intel machine code that your physical CPU can execute directly.
* **Networking**: The CPU executes the `new ServerSocket(5000)` code. This tells the macOS kernel: *"I want to claim network port 5000. Let me know if anyone tries to connect."* 
* **Blocking**: When the code hits `serverSocket.accept()`, the CPU doesn't just spin in an endless loop wasting power. Instead, the OS puts the Server thread to **sleep** (blocks it). The CPU is completely freed up to do other tasks until a network signal arrives.

#### Step 3: Starting the Client (`java ChatClient`)
* **What happens**: Another JVM process is launched in a separate terminal. It loads `ChatClient.class`.
* **Connecting**: The CPU hits `new Socket("localhost", 5000)`. It asks macOS to connect to port 5000. 
* **TCP Handshake**: The OS sends a network signal (even though it's on the same machine, it routes through a virtual network interface). It performs a "TCP 3-way handshake" (SYN, SYN-ACK, ACK) at the operating system level to establish a reliable connection pipeline.

#### Step 4: The Server Wakes Up
* **What happens**: macOS receives the network signal on port 5000. It says, *"Ah! The ChatServer process was waiting for this!"*
* **Context Switch**: The OS wakes up the Server thread. The CPU resumes executing code right where it left off at `accept()`, returning a shiny new `Socket` object representing the connection to the Client.
* **Multithreading**: The server creates a `new ClientHandler(clientSocket)` and calls `.start()`. The OS creates a **brand new thread** for this specific client. 
* **Parallelism**: If you have a multi-core processor, the main Server thread (waiting for new clients) and the ClientHandler thread (listening to Alice) might literally be executing on two completely different CPU cores at the exact same physical millisecond.

#### Step 5: Sending a Message
* **What happens**: In Terminal 2, you type "Hello Bob!" and press Enter.
* **Input handling**: When you press Enter, the physical keyboard sends an electrical interrupt to the CPU. The OS handles the interrupt, reads the keystroke, and passes it to the Client JVM, waking up `System.in.readLine()`.
* **Transmission**: The Client converts the string "Hello Bob!" into raw bytes and writes it to the socket. The OS breaks this data into small TCP packets, adds routing headers, and fires them across the internal network.
* **Reception**: The Server's `ClientHandler` thread for Alice—which was put to sleep waiting for data—is woken up by the OS. It reads the bytes, converts them back to a String, and loops through the `clients` list.
* **Broadcasting**: For every other client in the list (like Bob), the Server writes the string to their respective sockets. The OS transports the packets to Bob's JVM, which prints it to Bob's terminal screen.

#### Step 6: Disconnecting (`/quit`)
* **What happens**: Bob types `/quit`.
* **Client shutdown**: The Client code sees `/quit`, breaks its loop, and calls `socket.close()`. The OS sends a special TCP "FIN" (finish) packet to the server to gracefully close the pipeline. The Client JVM shuts down, and macOS reclaims its RAM.
* **Server cleanup**: The Server's `ClientHandler` for Bob tries to read the socket but receives an "End of File" (EOF) signal because the socket was closed. It catches this, removes Bob from the `clients` list, and the ClientHandler thread gracefully terminates.

In summary, your code acts as a director, telling the JVM how to orchestrate threads and memory. The JVM translates this to the OS, which heavily relies on hardware interrupts to put the processor to sleep when waiting for typing or network data, and rapidly wakes it up the microsecond an event occurs!

Phase 9: Features Walkthrough
The chat system has now been upgraded with advanced "Product-Level" features, moving it far beyond a simple broadcast server! In addition, critical security rules were introduced based on your feedback.

What Was Added
1. Robust Architecture & Security 🛡️
ConcurrentHashMap: In ChatServer.java, the List<ClientHandler> was replaced with a thread-safe mapping (ConcurrentHashMap<String, ClientHandler>). This enables O(1) lookups for direct messaging and natively handles duplicate usernames.
Server-Side Spoofing Prevention: A malicious client cannot pretend to be someone else anymore. ClientHandler.java now completely ignores the username field in incoming messages and overwrites it with the authenticated this.username associated with that specific TCP socket before processing.
Duplicate Username Validation: If a new client tries to JOIN with a username that is already in the map, the server responds with a strict JSON Command.ERROR message and forces a disconnect.
2. Private Messaging 🤫
Command: /msg <username> <message>
Implementation: The ChatClient parses this command and sends a Command.PRIVATE JSON object with a new targetUser property. The ChatServer uses ConcurrentHashMap.get(targetUser) to route the message only to the intended recipient's socket.
Output: (Private from Bob): Hello!
3. Online User List 👥
Command: /users
Implementation: The ChatClient sends a Command.USERS JSON object. The server dynamically pulls clients.keySet(), formats a string, and sends it back to the requesting client as a system message.
Output: *** Online users: Alice, Bob ***
4. Message Timestamps 🕒
Implementation: A new timestamp string field was added to the Message model. The ClientHandler generates a standardized HH:mm:ss timestamp (e.g., using LocalTime.now()) the moment a message arrives on the server. The client parses this and prepends it to the output.
Output: [15:14:42] Alice: Hello world
